# Refactor Plan: Monolith Budget Optimization

## Problem

Several codebase files exceed their `monolith_budget.toml` SLOC ceilings. Per the monolith budget discipline (Plan 2026-05-18-006), any growth beyond the ceiling requires a rationale update in the same PR. Currently:

| File | Current SLOC | Ceiling | Delta | Action Required |
|------|-------------|---------|-------|-----------------|
| `cli/_publish_helpers.py` | 797 | 660 | +137 | Extract lease/throttle or reconciler logic |
| `cli/generate_backlink_text.py` | 828 | 560 | +268 | Split into engine + shell modules |
| `publishing/adapters/__init__.py` | 767 | 600 | +167 | Move verify helpers to separate module |
| `content/fetch.py` | 527 | 250 | +277 | Extract more utilities to `_html_utils.py` |
| `cli/phase0_seal.py` | 572 | 465 | +107 | Already partially extracted; move show/verify/reseal helpers |
| `cli/plan_check.py` | 305 | 260 | +45 | Already extracted to sub-modules; minor cleanup |

## Analysis

Each file follows a pattern where a "thin shell" should own I/O/exit code handling while core logic lives in extracted modules. The codebase has precedent for this pattern (e.g., `plan_backlinks/_engine.py`, `validate/engine.py`, `_report_engine.py`).

### Optimization Opportunities

1. **`cli/_publish_helpers.py`** (R4 hard-fail: +137)
   - Already extracted from `publish_backlinks.py`
   - Extract `_run_reconciler` to `events/reconciler.py` (cross-ref says it exists)
   - Extract throttle cluster (`_load_throttle_config`, `_sleep_with_throttle`, `_medium_throttle_sleep`) to `_throttle.py`

2. **`cli/generate_backlink_text.py`** (R4 hard-fail: +268)
   - Split into `cli/generate_backlink_text/engine.py` (pure compute) + shell (`generate_backlink_text.py`)
   - Move `_read_candidates`, `_validate_candidate`, `_validate_generated_text`, `_run_dry_run`, `_run_generate` to engine

3. **`publishing/adapters/__init__.py`** (R4 hard-fail: +167)
   - Move `_verify_*_live` functions and shared helpers (`_ok_result`, `_timeout_result`, etc.) to `publishing/adapters/_verify.py`
   - Registry dispatch block stays minimal

4. **`content/fetch.py`** (R4 hard-fail: +277)
   - Extract cache utilities to `_cache.py`
   - Move `_check_once` to a separate `_fetch_core.py`

5. **`cli/phase0_seal.py`** (R4 hard-fail: +107)
   - Move `_handle_show`, `_handle_verify`, `_handle_reseal`, `_handle_verify_hook` to `_seal_handlers.py` (partially exists as `_seal_init.py`)

6. **`cli/plan_check.py`** (R4 hard-fail: +45)
   - Already fully extracted; may need to move some handler logic back into the flow

## Extraction Order

1. `publishing/adapters/__init__.py` → `_verify.py` (simplest, no cross-module deps)
2. `cli/_publish_helpers.py` → `_throttle.py`, `events/reconciler.py` integration
3. `content/fetch.py` → `_cache.py`, `_fetch_core.py`
4. `cli/generate_backlink_text.py` → `engine.py`
5. `cli/phase0_seal.py` → `_seal_handlers.py`

## Constraints

- `monolith_budget.toml` must be updated in the SAME PR as any file growth
- Each `rationale` field must be ≥80 chars explaining current shape + expected settling
- Git blame is the defense against silent regrowth
- Follow existing patterns: shell imports from engine, engine never touches stdout/stderr
