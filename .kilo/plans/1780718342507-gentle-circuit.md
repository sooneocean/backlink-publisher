## 2. 低內聚問題識別

### 2.1 巨大檔案 (高風險)

| 檔案 | SLOC | 問題 |
|------|------|------|
| `cli/phase0_seal.py` | 465 (ceiling) | 6 個子指令 + 通用邏輯混雜 |
| `cli/_publish_helpers.py` | 480 (ceiling) | 出版循環 + throttle + canary + checkpoint 邏輯混合 |
| `publishing/adapters/__init__.py` | 330 (ceiling) | 註冊 22 個平台、dispatcher、setup 驗證全包 |
| `publishing/registry.py` | 534 | Publisher ABC + 註冊表 + 並行 dict + manifest helper |
| `events/_project_reducers.py` | 550 (ceiling) | 3 個 reducer + 輔助函式混雜 |

### 2.2 功能重疊/模糊邊界

1. **`_publish_helpers.py`** 與 **`_publish_cli.py`** 邊界不清
   - `_publish_helpers.py` 持有 argparse、checkpoint ops、出版循環
   - `_publish_cli.py` 只抽出 argparse 和 checkpoint ops
   - 剩餘出版循環邏輯 (throttle, canary, verify, checkpoint) 仍在 `_publish_helpers.py`

2. **`publishing/registry.py`** 職責過多
   - Publisher ABC 定義
   - RegistryEntry dataclass
   - `_REJECTED_PLATFORMS` 拒絕平台地圖
   - `_AUTH_TYPE_*` 認證類型映射
   - `_REJECTION_*` 補償映射
   - 6 個 manifest helper 函式
   - `dispatch()` 函式

3. **`cli/phase0_seal.py`** 子指令混合
   - `init`, `verify`, `reseal`, `show` 每個子指令邏輯都在同一檔案

## 3. 耦合問題分析

### 3.1 層級循環依賴

主要的循環依賴鏈：
```
publishing.registry → _manifest_types
publishing.adapters.__init__ → registry.register (side effect)
publishing.adapters.__init__ → _manifests (每個平台的宣告)
registry → _registry_manifest → _REGISTRY (讀取)
registry → _registry_dispatch → _REGISTRY (讀取)
```

### 3.2 跨層存取過多

| 模組 | 跨層存取 | 問題 |
|------|---------|------|
| `webui_app/__init__.py` | `_REGISTRY`, `channel_status`, `canary/store`, `browser_publish` | 啟動時聚合過多外部依賴 |
| `publishing/adapters/__init__.py` | 47 個 adapter import + manifest import | 單一檔案責任過重 |
| `cli/_publish_helpers.py` | events store, linkcheck, canary, checkpoint | 出版循環邏輯廣泛依賴 |

## 4. 模組化拆分建議

### 4.1 建議 1: 拆分 `cli/_publish_helpers.py`

**目標**: 分離出版循環核心邏輯，降低與事件/驗證/金絲雀的耦合

```text
cli/publish_backlinks/
├── __init__.py           # main() only (199 行)
├── _engine.py            # 移入循環核心: _publish_one_row, run_publish_loop
├── _helpers_verify.py    # 驗證相關: _do_verify, _check_row_reachability
├── _helpers_canary.py    # 金絲雀相關: _canary_gate
├── _helpers_throttle.py  # 節流相關: _sleep_with_throttle, _medium_throttle_sleep
├── _helpers_checkpoint.py # Checkpoint 相關: _try_update_ckpt_failed
└── _publish_cli.py       # argparse + 持久化操作 (現有)
```

**優點**:
- 循環核心邏輯與跨模組依賴解耦
- 可單獨測試驗證邏輯
- 減少 `__init__.py` 的 SLOC 壓力

### 4.2 建議 2: 拆分 `publishing/registry.py`

**目標**: 分離 ABC、註冊表狀態、manifest helper、dispatch 邏輯

```text
publishing/registry/
├── __init__.py           # re-export 所有符號
├── _abc.py               # Publisher ABC
├── _state.py             # _REGISTRY, RegistryEntry, _REJECTED_PLATFORMS
├── _dofollow_meta.py      # dofollow 相關: _AUTH_TYPE_*, _REFERRAL_VALUE_*
├── _manifest_helper.py    # ui_meta, bind_descriptors, policy, visibility 等讀取
└── _dispatch.py          # dispatch() 函式
```

**優點**:
- Publisher ABC 變成可單獨測試
- Manifest helper 與 dispatch 邏輯解耦
- 降低巨大檔案風險

### 4.3 建議 3: 拆分 `events/_project_reducers.py`

**目標**: 每個 reducer 獨立模組

```text
events/_reducers/
├── __init__.py           # re-export 所有 reducer
├── checkpoint.py         # _project_checkpoint (約 180 行)
├── history.py            # _project_history (約 180 行)
├── drafts.py             # _project_drafts (約 180 行)
└── _helpers.py           # _emit_confirmed_* 輔助函式
```

**優點**:
- 降低單一 reducer 的維護難度
- 清晰的職責邊界
- 可獨立測試每個 reducer

### 4.4 建議 4: 拆分 `cli/phase0_seal.py` 為指令套件

**目標**: 每個子指令成獨立模組

```text
cli/phase0_seal/
├── __init__.py           # main() 路由分派
├── _init.py              # init 指令邏輯
├── _verify.py            # verify 指令邏輯
├── _reseal.py            # reseal 指令邏輯
├── _show.py              # show 指令邏輯
└── _shared.py            # 共享輔助函式
```

**優點**:
- 指令邏輯隔離
- 降低單一檔案複雜度
- 便於未來新增子指令

### 4.5 建議 5: 拆分 `publishing/adapters/__init__.py`

**目標**: 將 adapter 註冊移至獨立套件

```text
publishing/adapters/
├── __init__.py           # publish(), verify_adapter_setup(), re-export
├── registry.py           # 所有 register() 呼叫移入
├── base.py               # AdapterResult, BaseAdapter
└── {platform}/           # 每個平台獨立子套件 (選擇性)
    ├── __init__.py       # 單個 adapter 類
    ├── api.py            # API 客戶端
    └── ui.py             # 平台特定邏輯
```

**優點**:
- 新增平台不需修改 `__init__.py`
- 單一平台問題不影響其他平台
- 模組載入時間減少

## 5. 實施優先序

| 優先級 | 項目 | 理由 |
|--------|------|------|
| P0 | `cli/_publish_helpers.py` | 目前 SLOC 20+SLOC 壓力，造成 `publish_backlinks/__init__.py` 複雜 |
| P1 | `publishing/registry.py` | 職責過多，影響整個發布層架構 |
| P2 | `events/_project_reducers.py` | 單一檔案責任過重，影響事件投影穭 |
| P3 | `cli/phase0_seal.py` | 子指令邏輯混合，未來維護性差 |
| P4 | `publishing/adapters/__init__.py` | 可考慮為 Phase-2/3 規劃 |

## 6. 風險與緩解

| 風險 | 緩解方案 |
|------|---------|
| 測試會破壞 | 採用 re-export 模式保持現有 import 路徑 |
| import cycle | 使用 TYPE_CHECKING 懶載入 + 適當模組層級 |
| SLOC 上升 | 同步更新 `monolith_budget.toml` |
| 模組過多 | 採用套件內 `__init__.py` 重新匯出，保持單一進入點 |

## 7. 後續發展效益

### 7.1 架構清晰度
- 服務層級邊界明確 (CLI → 核心邏輯 → 發布層 → 平台)
- 單一職責原則 (SRP) 獲得更好落實

### 7.2 擴展性提升
- 新增平台僅需在 `adapters/registry.py` 新增 `register()` 呼叫
- 新增 CLI 子指令獨立成模組
- 新增 reducer 不影響現有程式碼

### 7.3 測試改善
- 單元測試可聚焦於單一模組
- Mock patch 點更明確
- 跨模組測試更容易隔離

### 7.4 維護效率
- 問題定位範圍縮小
- 程式碼審查焦點清楚
- Git blame 更精確反映變更原因
